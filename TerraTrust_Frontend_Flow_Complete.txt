SECTION 1: WHAT THIS DOCUMENT NORMALISES
================================================================================

The current SRS v3.1, Backend System Design v3.1, and FDD v3.1 already define
the core screen set, primary API contracts, and high-level navigation model.
This document now serves as the detailed navigation companion that resolves the
remaining cross-document ambiguities and makes the interaction rules explicit.

This document normalises:
  - exact navigator hierarchy (root stack, tab stacks, modal flows)
  - which screens are stack pushes versus full-screen modals
  - header and back-navigation rules per screen
  - CTA destinations from Home, Land, Audit, and Profile flows
  - polling, deep-link, notification, empty-state, and error-state behaviour
  - maintenance detection via GET /api/v1/status, while /health remains the
    liveness probe for infrastructure

Canonical clarifications applied here:
  - WalletRecoveryScreen belongs to the Profile stack. It is not a global
    utility modal, although the tab bar is hidden during that flow.
  - AuditStartScreen is always launched as a full-screen modal on the
    RootNavigator, regardless of whether the user starts from HomeScreen,
    LandListScreen, or LandDetailScreen.
  - GET /api/v1/credits/balance is the shared credits endpoint and returns the
    farmer's current balance plus paginated audit history.

================================================================================
SECTION 2: NAVIGATION ARCHITECTURE — COMPLETE STRUCTURE
================================================================================

The app uses THREE navigator types from React Navigation v7:

  1. RootNavigator (Native Stack) — top-level router between auth and main app
  2. MainTabNavigator (Bottom Tabs) — the primary navigation after login
  3. Sub-stack navigators — one per tab to allow deep navigation per feature

--------------------------------------------------------------------------------
2.1 ROOT NAVIGATOR
--------------------------------------------------------------------------------

RootNavigator is a Native Stack Navigator. It contains:

  STACK A — AuthStack (shown to unauthenticated users)
    Screens: SplashScreen, LoginScreen, OTPScreen, KYCScreen, OnboardingScreen

  STACK B — MainStack (shown to authenticated + KYC-complete users)
    Contains: MainTabNavigator as the root
    Plus modal screens pushed above tabs (full-screen modals):
      - ZoneNavigationScreen     (full-screen modal during active audit)
      - ARCameraScreen           (full-screen modal during active audit)
      - ManualMeasureScreen      (full-screen modal during active audit)
      - TreeResultScreen         (full-screen modal during active audit)
      - AuditCompleteScreen      (full-screen modal during active audit)
      - AuditStatusScreen        (full-screen modal after submission polling)
      - DocumentUploadScreen     (full-screen modal for adding land)
      - BoundaryConfirmScreen    (full-screen modal for confirming boundary)
      - ManualUploadGuideScreen  (full-screen modal for fallback land upload)
      - NotificationsScreen      (full-screen modal from header bell icon)
      - MaintenanceScreen        (full-screen modal, blocks all other screens)

  STACK C — Maintenance override
    MaintenanceScreen replaces everything if the backend returns a
    maintenance flag. No other screen is accessible.

--------------------------------------------------------------------------------
2.2 MAIN TAB NAVIGATOR (BOTTOM TABS)
--------------------------------------------------------------------------------

The bottom tab bar is ALWAYS visible when the farmer is in the main app,
EXCEPT during active AR audit screens (those are full-screen modals with
their own back/close button and no tab bar shown).

The tab bar has FOUR tabs:

  TAB 1 — Home (house icon)
    Stack: HomeStack
    Screens in stack:
      - HomeScreen (root of this tab)
      - CreditHistoryScreen (pushed from HomeScreen)
      - LandDetailScreen (pushed from HomeScreen land card tap)
      - EditLandNameScreen (pushed from LandDetailScreen)

  TAB 2 — My Lands (leaf/field icon)
    Stack: LandStack
    Screens in stack:
      - LandListScreen (root of this tab)
      - LandDetailScreen (pushed from LandListScreen card tap)
      - EditLandNameScreen (pushed from LandDetailScreen)
      NOTE: AuditStartScreen is not part of LandStack. It is opened as a
        full-screen modal on the RootNavigator from LandDetailScreen or
        LandListScreen, and the rest of the audit flow stays on that
        RootNavigator modal stack above the tab bar.

  TAB 3 — History (clock/chart icon)
    Stack: HistoryStack
    Screens in stack:
      - CreditHistoryScreen (root of this tab)
      NOTE: This is the same CreditHistoryScreen as the Home tab links to.
            Reuse the same screen component. Tab just navigates directly to it.

  TAB 4 — Profile (person icon)
    Stack: ProfileStack
    Screens in stack:
      - ProfileScreen (root of this tab)
      - SettingsScreen (pushed from ProfileScreen)
      - WalletRecoveryScreen (pushed from ProfileScreen)

TAB BAR VISUAL SPECS:
  Active icon + label colour:   #2F855A (Forest Green)
  Inactive icon + label colour: #A0AEC0 (Disabled Grey)
  Background:                   #FFFFFF (Card White)
  Border top:                   1px solid #E2E8F0 (light grey)
  Label font:                   Roboto Regular, 11sp
  Icon size:                    24px
  Tab bar height:               56px + bottom safe area inset
  Notification badge:           Red dot on Profile tab when wallet recovery
                                is pending (derived from
                                GET /api/v1/auth/me.wallet_recovery_status).
                                Red dot on Home tab when a
                                background-sync audit upload completes.

--------------------------------------------------------------------------------
2.3 HEADER BAR RULES
--------------------------------------------------------------------------------

Each screen must define its header bar. Rules are as follows:

DEFAULT HEADER (most screens):
  - Back arrow on left (navigates back in stack)
  - Screen title centred
  - Background: #FFFFFF
  - Title font: Roboto Bold 18sp, colour #2D3748

NO HEADER (full-screen screens):
  - SplashScreen: no header
  - LoginScreen: no header (logo shown in body)
  - OTPScreen: custom back arrow in body, no system header
  - ARCameraScreen: no header (camera is full-screen)
  - ZoneNavigationScreen: no header (map is full-screen)
  - MaintenanceScreen: no header

HEADER WITH RIGHT-SIDE ACTION:
  - HomeScreen: Bell icon (🔔) on right. Tapping opens NotificationsScreen.
                Show red dot badge on bell if there are unread notifications.
  - LandListScreen: No right-side action.
  - ProfileScreen: No right-side action.

NO BACK BUTTON (screens where going back is not allowed):
  - HomeScreen (root tab — back exits app with Android back)
  - LandListScreen (root tab)
  - CreditHistoryScreen when accessed via History tab (root tab)
  - ProfileScreen (root tab)
  - KYCScreen (farmer must complete KYC, cannot go back to OTP)
  - AuditCompleteScreen (cannot go back into scanning after submitting)
  - AuditStatusScreen (cannot interrupt polling)

SCREEN 1: SplashScreen
--------------------------------------------------------------------------------
File:       src/features/auth/screens/SplashScreen.tsx
Header:     None
Back:       None

What farmer sees:
  - TerraTrust logo centred on #F7FAFC background
  - Lottie spinning_leaf.json animation below logo
  - No text. Pure visual load state.

Logic (runs silently, farmer just sees the animation):
  Step 1: Check Firebase currentUser.
  Step 2: If no Firebase user → navigate to LoginScreen (replace, no back).
  Step 3: If Firebase user exists:
            Refresh ID token.
            Call GET /api/v1/auth/me.
  Step 4: If /auth/me returns 200:
            If kyc_completed is false → navigate to KYCScreen (replace).
            If kyc_completed is true:
              Check if this is the very first time the main app is loaded
              (MMKV key 'onboarding_complete' is missing or false).
              If onboarding not done → navigate to OnboardingScreen.
              If onboarding done → navigate to HomeScreen (tab navigator, replace).
  Step 5: If /auth/me returns 401 or network error:
            Clear Firebase session.
            Navigate to LoginScreen.

Timeout: If /auth/me takes more than 10 seconds, navigate to LoginScreen
         with banner: "Connection issue. Please log in again."

SCREEN 3: OTPScreen
--------------------------------------------------------------------------------
File:       src/features/auth/screens/OTPScreen.tsx
Header:     Custom in-body back arrow (top left). Tapping returns to LoginScreen.
Back:       Custom back, not system header.

What farmer sees:
  - "Enter the 6-digit code sent to +91XXXXXXXXXX"
    (last 4 digits of number shown, rest masked: +91 XXXXXX1234)
  - 6 individual digit input boxes in a row
  - "Resend in 28s" countdown timer
  - After timer reaches 0: "Resend OTP" link replaces timer text
  - "Verify" primary button (disabled until all 6 boxes filled)

Behaviour:
  - Each box accepts one digit. Auto-focuses next box.
  - Backspace on an empty box focuses the previous box.
  - Paste of 6 digits auto-fills all boxes.
  - "Resend OTP" calls signInWithPhoneNumber again with same number.

On "Verify" tap:
  - Disable button, show loading inside button.
  - Call confirmation.confirm(code) from Firebase.
  - On success:
      Call GET /api/v1/auth/me.
      If wallet_address is null:
        Run silent wallet creation (ethers.js → react-native-keychain).
        Call POST /api/v1/auth/register-wallet with public address only.
        Refresh /auth/me state in Redux.
      If kyc_completed is false → navigate to KYCScreen.
      If kyc_completed is true:
        If onboarding_complete is false → navigate to OnboardingScreen.
        Else → navigate to HomeScreen (tab navigator, replace stack).
  - On failure: Show error below boxes. Clear all 6 boxes. Re-focus box 1.

SCREEN 4: KYCScreen
--------------------------------------------------------------------------------
File:       src/features/auth/screens/KYCScreen.tsx
Header:     None (no back — farmer must complete KYC)
Back:       None

What farmer sees:
  - Header text: "Complete Your Profile"
  - Subtitle: "This is a one-time setup. Your name must match your land document."
  - Input 1: Full Name (text input, floating label)
  - Input 2: Aadhaar Number (numeric, 12 digits, floating label, masked after entry)
  - Important note in a yellow card: "Enter your name exactly as written on
    your 7/12 land document. A mismatch will block land registration."
  - Why Aadhaar is requested: OTP proves the phone number, but TerraTrust also
    needs a one-time identity anchor to compare the KYC name with the 7/12
    owner name and to re-verify the same person during wallet recovery.
  - "Continue" primary button (disabled until both fields valid)
  - Privacy note at bottom: "Your Aadhaar number is hashed before storage and
    never stored as plain text."

Validation:
  - Full Name: not empty, only letters and spaces, min 2 characters.
  - Aadhaar: exactly 12 digits.

IMPORTANT — Security rule:
  Aadhaar number must NEVER be stored in Redux state, MMKV, logs, or
  screenshots. It exists only in React local state (useState) during form
  filling and in the POST request body. Once the API call completes,
  clear the local state.

On "Continue" tap:
  - Disable button, show loading.
  - POST /api/v1/auth/kyc with { full_name, aadhaar_number }.
  - On success:
      Update auth Redux slice: kycCompleted = true, name = full_name.
      If onboarding_complete is false → navigate to OnboardingScreen.
      Else → navigate to HomeScreen (tab navigator, replace stack).
  - On failure: Show error message in red. Re-enable button.

SECTION 5: BACK NAVIGATION RULES (COMPLETE)
================================================================================

Rule 1: Stack screens always have a back button.
Rule 2: Root tab screens (HomeScreen, LandListScreen, CreditHistoryScreen,
        ProfileScreen) have NO back button. Android physical back exits app
        (show "Press back again to exit" snackbar on first press).
Rule 3: Modal flows (land registration, audit) have a CLOSE (✕) button that
        discards the flow and returns to the tab that launched it.
Rule 4: KYCScreen has NO back — farmer must complete KYC.
Rule 5: AuditCompleteScreen has NO back — cannot return to scanning after review.
Rule 6: AuditStatusScreen has NO back while status is PROCESSING /
  CALCULATING / READY_TO_MINT — Android back minimises app.
  Back is available only after a terminal state is shown
  (MINTED / COMPLETE_NO_CREDITS / FAILED).
Rule 7: OnboardingScreen has no back. Farmer can skip.
Rule 8: MaintenanceScreen blocks all navigation. No back possible.
Rule 9: DocumentUploadScreen close (✕) cancels the entire land registration flow.
        Within the flow: BoundaryConfirmScreen back → DocumentUploadScreen.
        ManualUploadGuideScreen back → DocumentUploadScreen.

  terratrust://audit/{audit_id} → AuditStatusScreen
  terratrust://land/{land_id} → LandDetailScreen

  Respect Retry-After header. No aggressive retry loop.

  Banner: "Server issue. Please try again in a few minutes."
  (Already handled by Axios interceptor in api.ts)

All screens — 401 token expired:
  Firebase signOut. Navigate to LoginScreen.

SECTION 11: BACKEND API CLARIFICATIONS USED BY THIS SPEC
================================================================================

The following backend contracts are the canonical interfaces used by this
navigation spec. PATCH /api/v1/land/{land_id} and
POST /api/v1/auth/recover-wallet already exist in the SRS/BSDD and are
retained here for clarity. POST /api/v1/land/fetch-boundary is the manual-map
processing variant of the boundary endpoint. GET /api/v1/status is the
maintenance-mode probe; /health remains the infrastructure liveness endpoint.

  PATCH /api/v1/land/{land_id}
    Purpose: Update farm name (editable by farmer).
    Sends:   { farm_name: string }
    Returns: { status: 'success', land_id: UUID, farm_name: string }

  POST /api/v1/auth/recover-wallet
    Purpose: Submit wallet recovery request (SRS Appendix item 9).
    Sends:   { new_wallet_address: string }
    Returns: { status: 'pending', message: string }

  POST /api/v1/land/fetch-boundary
    Purpose: Process a farmer-uploaded government parcel map in the manual
             fallback flow.
    Sends:   multipart/form-data with map_image, survey_number, district,
             taluka, village, state
    Returns: Standard boundary success payload with boundary_source: 'MANUAL'

  GET /api/v1/status
    Purpose: Maintenance mode detection and app bootstrap status.
    Returns: { maintenance: boolean, message: string | null }
    Called:  By SplashScreen on boot to check if maintenance banner is needed.
