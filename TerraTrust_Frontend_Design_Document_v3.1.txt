  design that uses Google satellite tiles across all map screens.

   Section 12 (Offline Strategy) previously stated "Document OCR requires
   backend Tesseract server." Updated to reflect that the backend now uses
  Google Cloud Vision API (changed in backend document v3.1). The frontend
   behaviour is identical — OCR still happens on the server, farmer still
   sees the same extracted fields. Only the server-side engine changed.

10. Authentication model corrected.
  Supabase Auth is removed from the mobile app. Firebase Authentication is
  now used only for phone OTP sign-in, and the app calls FastAPI with a
  Firebase ID token after login. Supabase remains backend-only for
  PostgreSQL, PostGIS, and file storage.

11. Certificate URL source clarified.
  Audit certificates are still addressed on-chain as ipfs://CID, but the app
  opens backend-returned Pinata gateway URLs in the browser for farmer-facing
  certificate viewing.

WHAT CHANGED FROM VERSION v2.1 (carried over from v3.0)
--------------------------------------------------------------------------------

The following changes were made in v3.0 and remain in effect:

1. Supabase Auth removed; Firebase restored for authentication only.
  Reason: Phone OTP sign-in now uses Firebase Authentication, while Supabase
  remains backend-only for PostgreSQL + PostGIS + Storage.
  Impact: Replace the old Supabase client-side auth calls with
  @react-native-firebase/app and @react-native-firebase/auth. All app data
  still flows through the FastAPI backend.

2. Land boundary fetching is now automated (3-layer system).
   Previous design had two modes: Draw Mode (manual polygon tap) and Walk Mode
   (GPS-based). Both are now REMOVED as primary methods.
   Reason: Official government land boundary must come from government records,
   not farmer-drawn polygons, to be accepted by carbon markets.
   New approach: OCR the 7/12 document → auto-fetch official boundary via
   BhuNaksha WMS API → farmer only confirms what is shown.
   Manual drawing retained only as last-resort fallback if all API layers fail.

3. AR Measurement System changed from single-tier to 3-tier.
   Previous design assumed all phones support ARCore Depth API.
   Reality: Not all Android phones have depth sensor hardware.
   New approach: Tier 1 (full depth AR), Tier 2 (motion-based AR), Tier 3
   (manual string measurement). App auto-detects which tier applies.

4. AR no longer measures tree height as primary source.
   Previous design: AR measures height using AR anchors (user taps base, tilts
   to top).
   New design: Height comes from NASA GEDI satellite data where available.
   AR height measurement retained as fallback only when GEDI has no data for
   that location.
   Reason: GEDI provides more consistent height measurements across the whole
   farm without farmer error.

5. Sampling zones are now satellite-guided, not arbitrary.
   Previous design: 3-5 zones placed based on compass directions (North, South,
   Center).
   New design: Backend runs Sentinel-2 NDVI analysis on the farm first. Zones
   are placed inside distinct vegetation density areas identified by satellite.
   Farmer follows a linear walking path (like Google Maps navigation), not
   random directions.

6. Redux persist storage changed from AsyncStorage to MMKV.
   AsyncStorage is slow and not encrypted. MMKV is 30x faster, synchronous,
  and supports optional encryption. Critical for AR scan data that must save
  instantly.

7. react-native-image-picker replaced by react-native-vision-camera for
   document scanning.
   Reason: Higher quality capture needed for OCR accuracy on 7/12 documents.

1.1 DESIGN PATTERN
------------------
The application follows two principles:

Atomic Design Methodology: Components are broken into Atoms (Button, Text,
Icon), Molecules (InputField, OTPBox), Organisms (LoginForm, TreeScanCard),
Templates, and Screens. This ensures reuse and consistency.

Feature-First Folder Structure: All code for a feature lives together.
For example, src/features/ar-audit/ contains the screens, Redux slice, and
utilities for the AR audit module. This scales better than grouping by file
type.

1.3 NAVIGATION ARCHITECTURE
---------------------------
The mobile app uses three navigator layers from React Navigation v7:

RootNavigator (Native Stack):
  - AuthStack for SplashScreen, LoginScreen, OTPScreen, KYCScreen,
    and OnboardingScreen.
  - MainAppStack for the tabbed application shell after successful auth.
  - Maintenance override that places MaintenanceScreen above all other flows
    whenever the backend signals maintenance mode.

MainTabNavigator (Bottom Tabs):
  - Home tab: HomeScreen as root plus stack pushes for LandDetailScreen and
    CreditHistoryScreen.
  - My Lands tab: LandListScreen as root plus LandDetailScreen and
    EditLandNameScreen.
  - History tab: CreditHistoryScreen as the root.
  - Profile tab: ProfileScreen as root plus SettingsScreen and
    WalletRecoveryScreen.

Modal flows above tabs:
  - Land registration: DocumentUploadScreen, BoundaryConfirmScreen,
    ManualUploadGuideScreen, LandRegistrationSuccessScreen.
  - Annual audit: AuditStartScreen, ZoneNavigationScreen, ARCameraScreen,
    ManualMeasureScreen, TreeResultScreen, AuditCompleteScreen,
    AuditStatusScreen.
  - Utility modals: NotificationsScreen and MaintenanceScreen.

Tab bar behavior:
  - Visible throughout normal browsing inside the tab stacks.
  - Hidden during land registration, audit, notifications modal,
    wallet recovery flow, and maintenance mode.
  - Active tint: #2F855A. Inactive tint: #A0AEC0.
  - Height: 56px plus safe-area inset.
  - Home tab may show a red dot for unread notifications.
  - Profile tab may show a red dot while wallet recovery is pending.

Header and back-navigation rules:
  - Root tabs have no back button.
  - SplashScreen, LoginScreen, OTPScreen, ARCameraScreen,
    ZoneNavigationScreen, and MaintenanceScreen use custom or no system header.
  - HomeScreen uses a bell icon as the right-side header action.
  - KYCScreen, AuditCompleteScreen after review begins, and AuditStatusScreen
    while the audit is non-terminal do not allow navigating back into the
    previous flow.

SECTION 3: MODULE 1 — AUTHENTICATION
================================================================================

3.1 OVERVIEW
------------
Authentication uses Firebase phone OTP. The farmer enters their Indian mobile
number, receives a 6-digit SMS code, and is logged in. No passwords. No email.
This is the only way any Indian farmer should be expected to log in.

After Firebase verifies the OTP, the app immediately calls
GET /api/v1/auth/me with the Firebase ID token. This bootstrap call tells the
app whether KYC is complete and whether a wallet already exists. The backend
verifies that Firebase ID token with Firebase Admin SDK before responding. The
app never talks directly to Supabase from the device.

KYC (Aadhaar + Name) is collected once at first login. The name collected here
is later cross-checked against the owner name extracted from their 7/12 land
document by OCR. The farmer should enter it as written on the document, but
backend validation uses fuzzy matching (>=80% similarity) to tolerate OCR and
spacing differences. Below that threshold, land registration is rejected.

When a new farmer logs in for the first time and has no wallet address on
record, the app silently generates a blockchain wallet. The farmer sees nothing
of this process. It happens in the background. This wallet receives their
Carbon Ton Tokens when they earn them.

-----------

SCREEN: OTPScreen

                     code sent to your number"
                     Timer showing "Resend in 28s" counting down to zero.

                     If KYC done and onboarding_complete is false:
                     navigate to OnboardingScreen.
                     If KYC done and onboarding_complete is true:
                     navigate to MainTabNavigator with HomeScreen selected.

SCREEN: KYCScreen
  What farmer sees:  Two input fields: Full Name, Aadhaar Number (12 digits).
                     Instruction text: "Enter your name exactly as written on
                     your land document (7/12 Extract)."
                     One button: "Continue"
  Why this matters:  The name entered here is matched against the OCR-extracted
                     owner name from the land document in Module 2. Mismatch

  Why Aadhaar is collected:
                     OTP proves control of the phone number, but not the
                     farmer's real-world identity. TerraTrust uses Aadhaar in
                     v3.1 as the one-time KYC identifier so the backend can
                     bind the account to a farmer identity, compare that KYC
                     name against the 7/12 owner name, and re-verify the same
                     person during wallet recovery.
  On submit:         POST /api/v1/auth/kyc to backend over HTTPS with the raw
                     Aadhaar number. Backend hashes Aadhaar with SHA-256 before
                     storage (never stored plain text) and saves only the hash
                     in the backend users table. Clear Aadhaar from local
                     component state immediately after the request resolves.
                     Navigate to OnboardingScreen on first successful KYC.

SCREEN: OnboardingScreen
  What farmer sees:  Three swipeable tutorial cards with dot indicators.
                     Card 1: land verification with official records.
                     Card 2: zone navigation and tree scanning.
                     Card 3: credits, history, and certificates.
                     "Next" button on cards 1 and 2.
                     "Get Started" button on card 3.
                     "Skip" link on cards 1 and 2.
  On complete:       Write onboarding_complete = true to MMKV and navigate to
                     MainTabNavigator with HomeScreen selected.

3.3 SILENT WALLET CREATION
---------------------------
Happens automatically after first OTP verification when GET /api/v1/auth/me
shows no wallet_address.
The farmer never sees this process and never interacts with it.

Step 1: ethers.js generates a random private key and derives the public wallet
        address from it.
Step 2: The private key is encrypted and stored in react-native-keychain, which
        uses the phone's hardware-backed secure storage (Android Keystore).
        The key never leaves the device. It is never stored in Redux state.
        It is never sent to any server.
Step 3: Only the public wallet address is sent to the FastAPI backend and
        stored in the users table under wallet_address.
Step 4: When the farmer earns Carbon Ton Tokens, the backend mints them to this
        wallet address. The farmer sees their balance in the Dashboard as a
        plain number, not as a blockchain address.

Code reference:
  // services/wallet.ts
  import { ethers } from 'ethers';
  import * as Keychain from 'react-native-keychain';

  export const createFarmerWallet = async (): Promise<string> => {
    const wallet = ethers.Wallet.createRandom();
    await Keychain.setGenericPassword(
      'wallet_private_key',
      wallet.privateKey,
      {
        service: 'terratrust_wallet',
        accessControl: Keychain.ACCESS_CONTROL.BIOMETRY_ANY_OR_DEVICE_PASSCODE,
      }
    );
    return wallet.address; // Only the public address is returned
  };

SECTION 4: MODULE 2 — LAND VERIFICATION
================================================================================

4.1 OVERVIEW
------------
This module verifies which land belongs to the farmer and stores its exact
official boundary. This happens once per land parcel. The boundary is then used
for two things: (1) geofencing during AR scans to confirm farmer is on their
own land, and (2) satellite pixel clipping to calculate biomass only within the
verified boundary.

The boundary MUST come from government records. Farmer-drawn polygons are not
acceptable for carbon credit verification by any carbon market standard. The
app auto-fetches the official boundary using a 3-layer system.

4.2 THE 3-LAYER BOUNDARY FETCHING SYSTEM
-----------------------------------------
These layers run in sequence. The app tries Layer 1 first. If it fails, it
tries Layer 2. If that also fails, it falls back to Layer 3. The farmer sees
the same result regardless of which layer succeeded — the boundary displayed on
a map for their confirmation.

LAYER 1 — Automatic via LGD API + BhuNaksha WMS (Primary, tried first)
  How it works:
  1. Farmer uploads photo of their 7/12 Extract or Record of Rights document.
  2. Image sent to backend. Google Cloud Vision API extracts:
     Survey Number, Owner Name, Village, Taluka, District, State.
  3. Backend calls Maharashtra LGD API (http://115.124.105.220/API/) with the
     extracted District and Taluka to get official LGD location codes.
     Endpoint used: POST /GetVillagesOfDistrictAndTaluka
     Returns: distcode, talukacode, village_code
  4. Backend constructs the gis_code from: state code + division code + taluka
     code + village code + survey number padded to 6 digits.
  5. Backend calls BhuNaksha WMS GetFeatureInfo endpoint:
     https://mahabhunakasha.mahabhumi.gov.in/WMS?SERVICE=WMS
       &VERSION=1.3.0&REQUEST=GetFeatureInfo
       &INFO_FORMAT=application/json
       &gis_code=[constructed_code]
       &[bounding box and pixel coordinates derived from farmer GPS]
  6. WMS returns GeoJSON with exact polygon coordinates of the survey number.
  7. GeoJSON sent back to app. Boundary shown on map.
  Coverage: All 36 districts of Maharashtra. Other major states have the same
  NIC BhuNaksha software deployed — state router in backend switches the base
  URL per state. Same logic, different domain per state.

  How it works:
  If Layer 1 WMS returns an error or times out, backend attempts to scrape the
  BhuNaksha portal using Selenium/Playwright with the same extracted fields.
  Result is still GeoJSON. App behaviour is identical.
  Used when: WMS endpoint is temporarily down or returns unexpected response.

LAYER 3 — Manual Guided Upload (Last Resort)
  How it works:
  If both Layer 1 and Layer 2 fail, or the farmer explicitly switches to
  manual map upload after rejecting an auto-fetched boundary, the app shows
  the ManualUploadGuideScreen.
  This screen guides the farmer step by step to download their own map from the
  government portal and upload it.
  Steps shown to farmer:
    Step 1: "Open bhunaksha.mahabhumi.gov.in" (shown as tappable link)
    Step 2: "Select your District, Taluka, Village from the menus"
    Step 3: "Find your Survey Number and tap Download"
    Step 4: "Come back here and upload the downloaded image"
  After farmer uploads the image, backend uses OpenCV to extract the polygon

  Result is still GeoJSON. Farmer confirms boundary the same way.
  Used when: State not yet integrated in state router, portal is down, or rural

SECTION 5: MODULE 3 — AR TREE SCANNING (AUDIT)
================================================================================

5.1 OVERVIEW
------------
Once per year, the farmer performs an audit. They walk to 3 sampling zones
on their land and scan trees at each zone. The measurements collected are:
  - Tree species (from TensorFlow Lite, on-device)
  - Trunk diameter in cm (from ARCore or manual string method)
  - GPS location of each tree
  - One evidence photo per tree

Tree HEIGHT is NOT collected by the farmer in the primary flow. Height comes
from NASA GEDI satellite data which the backend fetches automatically. AR height
measurement (tapping base and top with anchors) is used only as a fallback
when GEDI has no laser footprint data at that location.

5.2 HOW SAMPLING ZONES ARE DETERMINED
--------------------------------------
Before the farmer starts walking, the backend:
1. Fetches Sentinel-2 optical satellite imagery for the farm boundary.
2. Calculates NDVI (Normalized Difference Vegetation Index) for every pixel
   inside the boundary.
3. Classifies the parcel into three vegetation-density bands: high density,
  medium density, and low density.
4. Allocates 2-8 sampling circles across those density bands depending on
  farm size.
5. Places one sampling point at the centre of each returned zone.
6. Calculates a linear walking path through all returned zone points
  (shortest route through all points, like a transect).
7. Returns GPS coordinates of the zone centres to the app.

The number of zones and the plot radius scale with farm size:
  Up to 1 acre:    2-3 zones, 7 metre radius per zone
  1 to 3 acres:    3-4 zones, 9 metre radius per zone
  3 to 10 acres:   4-6 zones, 11 metre radius per zone
  Above 10 acres:  6-8 zones, 11 metre radius, split into 2 sessions on
                   different days. Progress saved between sessions.

Farmer scans every returned zone radius (shown as a circle on map).
Minimum 3 trees per returned zone. Maximum 5 trees per zone.
min_trees_required from the backend equals 3 x generated zone count.
Sparse parcel portions that do not receive a sampling circle are handled by
the satellite fusion engine and do not create extra scan obligations.

5.3 THE 3-TIER AR MEASUREMENT SYSTEM
--------------------------------------
The app checks ARCore availability once on startup and stores the result in
Redux (audit.arTier). This check determines which measurement method is used
for the entire audit session.

TIER 1 — Full Depth AR (Phones with hardware depth sensor support)
  Applicable to:    High-end phones above approx. Rs 15,000 that have a
                    Time-of-Flight (ToF) sensor or processors powerful enough
                    for dense depth estimation. ARCore Depth API returns
                    SUPPORTED_INSTALLED with high confidence depth maps.
  What farmer does: Hold phone 1-2 metres from tree trunk, facing the trunk
                    directly. Tap "Measure". Hold still for 3 seconds.
  What happens:     ARCore captures a 16-bit raw depth map. Kotlin code
                    identifies the cluster of 3D points that form the trunk in
                    the centre of the frame. RANSAC algorithm fits a 3D
                    cylinder to those points. Diameter = cylinder radius × 2.
  Validation:       If RANSAC cylinder fit error exceeds 5%, show prompt:
                    "Move closer to the tree and hold still, then tap Measure
                    again."
  Accuracy:         ±2-3 cm error.
  UI indicator:     Green badge shown after measurement: "High Precision"

TIER 2 — Motion-Based AR (ARCore phones without dedicated depth sensor)
  Applicable to:    Mid-range ARCore-compatible phones without a ToF sensor.
                    ARCore uses SLAM (Simultaneous Localization and Mapping)
                    to calculate depth from camera motion combined with
                    gyroscope and accelerometer data. No special hardware
                    needed — same ARCore Depth API, lower quality output.
  What farmer does: Hold phone 1-2 metres from trunk. Tap "Measure". Slowly
                    move phone left and right for 5 seconds.
                    App shows animated arrow: ← → to guide the motion.
  What happens:     SLAM builds a 3D model from the parallax between frames as
                    the phone moves. RANSAC cylinder fit runs on the resulting
                    point cloud. Same diameter extraction as Tier 1.
  Accuracy:         ±4-5 cm error.
  UI indicator:     Blue badge shown after measurement: "Standard Precision"

TIER 3 — Manual String Measurement (Any phone, no ARCore required)
  Applicable to:    Phones where ARCore returns UNSUPPORTED or NOT_AVAILABLE.
                    Budget phones below approx. Rs 8,000, very old devices.
  What farmer does: App shows ManualMeasureScreen with a diagram showing:
                    "Wrap a string or cloth around the tree at chest height.
                    Mark where the string meets itself. Lay it flat and
                    measure the length in centimetres."
                    Farmer enters this length (circumference) in a number input.
  What happens:     App calculates: diameter_cm = circumference_cm / 3.14159
                    This is scientifically identical to measuring with a girth
                    tape, which is the standard method used by professional
                    foresters worldwide.
  Accuracy:         Exact, provided farmer measures string correctly.
  UI indicator:     Grey badge: "Manual Measurement"

DETECTION CODE:
  // features/ar-audit/arTierDetector.ts
  export const detectARTier = async (): Promise<1 | 2 | 3> => {
    const support = await NativeModules.ARModule.checkDepthSupport();
    // Returns: 'FULL_DEPTH', 'SLAM_ONLY', or 'UNSUPPORTED'
    if (support === 'FULL_DEPTH') return 1;
    if (support === 'SLAM_ONLY') return 2;
    return 3;
  };

7.2 WHAT MUST BE PERSISTED
---------------------------
Use redux-persist with MMKV as the storage adapter.
MMKV must be used instead of AsyncStorage because:
  - 30x faster read/write speed
  - Synchronous API — tree scan data saves instantly when farmer taps "Save"
  - Optional encryption support when the key is sourced securely at runtime
  - Intended for persisted app state, not private keys or backend secrets
  - Current repo baseline uses react-native-mmkv 3.3.x on React Native 0.84.1
  - New Architecture stays enabled

The following state slices MUST be persisted to MMKV:

  audit.scannedTrees      — Most critical. If farmer scans 8 trees and the app
                            crashes or phone dies, all 8 trees must be there
                            when they reopen the app. Losing this data means the
                            farmer must redo the entire audit.
  audit.activeAuditId     — App must know an audit is in progress.

8.1 AXIOS CONFIGURATION
------------------------

  // services/api.ts
  import axios from 'axios';
  import Config from 'react-native-config';
  import auth from '@react-native-firebase/auth';

  const api = axios.create({
    baseURL: Config.API_BASE_URL,    // From .env via react-native-config
    timeout: 60000,                  // 60 seconds — satellite calls can be slow
  });

  // Auto-attach Firebase ID token to every request
  api.interceptors.request.use(async (config) => {
    const currentUser = auth().currentUser;
    if (currentUser) {
      const idToken = await currentUser.getIdToken();
      config.headers.Authorization = `Bearer ${idToken}`;
    }
    return config;
  });

  // Global response error handling
  api.interceptors.response.use(
    (response) => response,
    (error) => {
      if (error.response?.status === 401) {
        // Token invalid or expired — force re-authentication
        auth().signOut();
        navigateTo('LoginScreen');
      } else if (error.response?.status === 500) {
        // Server error — show maintenance banner
        showBanner('Server issue. Please try again in a few minutes.');
      } else if (!error.response) {
        // No internet — show offline banner
        showBanner('No internet connection. Your data is saved locally.');
      }
      return Promise.reject(error);
    }
  );

SECTION 10: NATIVE AR BRIDGE (KOTLIN)
================================================================================

10.1 PURPOSE
------------
React Native JavaScript code cannot directly call Google ARCore's depth and
point cloud APIs. A Native Module written in Kotlin acts as a bridge. The
Kotlin code does all ARCore work and exposes simple functions that JavaScript
calls as if they were regular async functions.

10.2 TYPESCRIPT INTERFACE
--------------------------

  // services/ar-bridge.ts
  interface ARMeasurementResult {
    diameter_cm: number;       // Calculated trunk diameter
    confidence: number;        // 0.0 to 1.0 — RANSAC fit quality
    tier_used: 1 | 2;         // 1 = hardware depth, 2 = SLAM motion
    point_count: number;       // Number of 3D points used in calculation
  }

  // Measures tree trunk diameter using ARCore
  export const measureTreeDiameter = async (): Promise<ARMeasurementResult> => {
    const result = await NativeModules.ARModule.measureCylinder();
    return JSON.parse(result);
  };

  // Checks what level of depth support this phone has
  export const detectARTier = async (): Promise<1 | 2 | 3> => {
    const support = await NativeModules.ARModule.checkDepthSupport();
    if (support === 'FULL_DEPTH') return 1;
    if (support === 'SLAM_ONLY') return 2;
    return 3; // UNSUPPORTED
  };

10.3 KOTLIN MODULE IMPLEMENTATION (ARModule.kt)
-------------------------------------------------

The ARModule.kt file must:

1. Extend ReactContextBaseJavaModule
   class ARModule(reactContext: ReactApplicationContext)
     : ReactContextBaseJavaModule(reactContext)

2. Override getName() to return "ARModule"
   This is the name used in NativeModules.ARModule in JavaScript.

3. Implement checkDepthSupport() with @ReactMethod annotation
   Check: ArCoreApk.checkAvailability(context)
   Check: session.isDepthModeSupported(Config.DepthMode.RAW_DEPTH_ONLY)
   Return string: 'FULL_DEPTH', 'SLAM_ONLY', or 'UNSUPPORTED'

4. Implement measureCylinder() with @ReactMethod annotation
   Step 1: Configure ARCore session:
           config.setDepthMode(Config.DepthMode.RAW_DEPTH_ONLY)
           RAW_DEPTH_ONLY is used, not AUTOMATIC, because raw depth gives
           more accurate values. Raw depth does not fill missing pixels, which
           means only confident depth measurements are used.
   Step 2: Access depth data:
           val depthImage = frame.acquireRawDepthImage16Bits()
   Step 3: Filter to centre region of frame (the reticle area, approximately
           the centre 30% of width and 60% of height).
   Step 4: Convert depth pixels to 3D world coordinates using camera intrinsics.
   Step 5: Run RANSAC cylinder fitting loop:
           - Randomly sample 2 points to define an axis direction
           - Randomly sample 1 point to define radius
           - Count how many remaining points fall within epsilon distance of
             the cylinder surface (these are "inliers")
           - If inlier count > best so far, update best cylinder
           - Repeat N times (N = 100 iterations minimum)
   Step 6: Calculate confidence = best_inlier_count / total_points
           If confidence < 0.7, return error: fit quality too low.
   Step 7: Return JSON:
           {"diameter_cm": X, "confidence": Y, "tier_used": 1or2,
            "point_count": Z}

5. Register ARModule in ARPackage.kt
   ARPackage must implement ReactPackage and return ARModule in
   createNativeModules().
   ARPackage must be added to getPackages() in MainApplication.kt.

Separate native hashing helper:
  HashModule.kt under java/com/terratrustar/security/ exposes
  sha256(base64Data: String) to JavaScript. It decodes the image bytes and uses
  java.security.MessageDigest.getInstance("SHA-256") to compute the digest.
  HashPackage.kt registers HashModule with React Native, and hash.ts wraps it
  for the TypeScript layer.

SECTION 11: SECURITY
================================================================================

THREAT: Farmer scans trees outside their registered land
PROTECTION: useGeofence performs point-in-polygon checks locally against the
cached official GeoJSON boundary on every GPS update during audit. Scanning
button disabled if outside boundary.
Message shown: "You appear to be outside your registered land. Please return
to your land to scan."
Backend revalidation still occurs during POST /api/v1/audit/submit-samples
using PostGIS ST_Contains() for every submitted tree point.

THREAT: Farmer uses a fake/mocked GPS location
PROTECTION: On audit start, Kotlin code checks:
  Settings.Secure.getInt(contentResolver,
    Settings.Secure.ALLOW_MOCK_LOCATION, 0)
If mock locations are enabled, show blocking screen:
"Please disable Mock Location in Developer Settings to use this app."
Audit cannot start until mock location is disabled.

THREAT: Evidence photo tampered with after upload
PROTECTION: SHA-256 hash of each evidence photo computed on-device before
upload. Hash sent to backend along with photo. Hash stored on blockchain.
Any post-upload modification to the photo produces a different hash that
no longer matches the on-chain record.
  // common/utils/hash.ts
  import { NativeModules } from 'react-native';
  const { HashModule } = NativeModules;

  export const hashPhoto = async (base64Data: string): Promise<string> => {
    return HashModule.sha256(base64Data);
  };

  // HashModule.kt decodes the base64 image bytes and hashes them using
  // java.security.MessageDigest with the SHA-256 algorithm.

THREAT: Private key exposure
PROTECTION: Private key stored ONLY in react-native-keychain using Android
Keystore hardware-backed encryption. Never stored in Redux state. Never stored
in AsyncStorage or MMKV. Never logged. Never sent to any server.

THREAT: Firebase ID token theft
PROTECTION: Firebase ID tokens are short-lived and refreshed by the Firebase
Auth SDK. The app requests the token only when making backend API calls.
Only derived auth state is persisted in MMKV; Supabase credentials and service
keys never exist on the device.

THREAT: App running on rooted phone
PROTECTION: react-native-device-info checks isRooted(). If rooted device
detected, show warning: "This device appears to be rooted. For your security,
some features may not work correctly." App continues — not blocked, because
some legitimate users have custom ROMs.

APPENDIX A: ALIGNMENT CLARIFICATIONS (APRIL 2026)
================================================================================

1. Boundary reject flow
  Rejecting a boundary does not discard the OCR result. The screen must offer
  "Try Automatic Fetch Again", "Use Manual Map Upload", and
  "Retake Document Photo". Returning to the original document capture flow is
  always an explicit choice.

2. Species model contract
  The on-device TFLite model uses RGB 224x224 input and outputs the 11
  approved species plus NOT_APPROVED. The saved tree payload must include
  species_source with one of these values:
    MODEL_AUTO for top approved label with species_confidence >= 0.80
    MODEL_CONFIRMED for top approved label with 0.60 <= species_confidence < 0.80
                   after explicit farmer confirmation
    MANUAL_SELECTED for farmer-chosen approved label after a low-confidence
                    or NOT_APPROVED result
  The backend accepts only approved species labels and validates that
  species_source is consistent with species_confidence.
  Model provenance: the classifier is trained offline from a curated labeled
  species image dataset, exported as a versioned TFLite artifact, and shipped
  in src/assets/tflite/species_model.tflite. The display label
  order/class mapping is fixed for each model artifact and must be bundled
  with that same model version; the app must not hardcode a different label
  order than the shipped asset.
  The display label
  "Indian Rosewood" in the UI refers specifically to Dalbergia sissoo,
  with Shisham / North Indian Rosewood treated as display aliases for the
  same approved species.

3. GPS quality rules
  Zone arrival requires distance <= 10m and gps_accuracy_m <= 20.
  Show a weak-signal warning when gps_accuracy_m > 20.
  Block saving a tree when gps_accuracy_m > 30.

4. Zone-level GEDI fallback and parcel-wide fallback
  GET /api/v1/audit/zones computes gedi_available per zone.
  If gedi_available=false for a zone, show the height-measurement control for
  every tree in that zone.
  If gedi_available=true for a zone, hide height capture and let the backend
  resolve height from the nearest valid GEDI sample inside that zone.
  If GEDI is unavailable for every zone in the parcel, exclude the GEDI
  feature from the parcel model for that audit.

5. COMPLETE_NO_CREDITS handling
  AuditCompleteScreen must treat COMPLETE_NO_CREDITS as a terminal success
  state, not as an error. Show the audit summary and the reason string, but
  no minting animation and no blockchain transaction hash.

6. Photo hashing rule
  Capture the tree trunk at the moment of measurement, resize to a maximum of
  1280×960 pixels, strip EXIF metadata, compress at JPEG quality 85, then hash
  the exact decoded bytes of evidence_photo_base64. Do not hash an earlier
  camera frame and then re-encode it.

7. Pagination behavior
  LandListScreen and CreditHistoryScreen load page 1 first and append more on
  scroll when has_more=true. HomeScreen uses the first page and displays only
  the first two history items in the preview card.

8. Rate limiting and 429 handling
  On HTTP 429, show a plain-language retry message and back off using the
  Retry-After header. Do not loop aggressive polling. audit/result polling
  remains every 5 seconds only while the audit is in a non-terminal state.

9. Wallet recovery policy
  The production product requires an admin-assisted wallet rotation flow.
  After OTP + KYC re-verification on a new device, the app creates a
  replacement wallet locally and the backend performs the documented recovery
  workflow for future issuance. Already-issued on-chain balances remain at the
  original wallet in v3.1 and must be disclosed clearly in the UI. This is a
  production-launch dependency, not an optional future enhancement.

10. Aadhaar compliance
  Aadhaar is collected only because OTP proves phone possession, while the
  platform also needs a one-time identity anchor for owner-name matching and
  wallet-recovery re-verification.
  It is not used for biomass calculation, credit minting, or routine app use.
  Plain Aadhaar is allowed only in volatile memory during form submission.
  It must never be written to Redux, MMKV, screenshots, analytics, or logs.
  Backend storage is hash-only.
  Production launch requires legal review against India's DPDPA 2023.

11. Monitoring, backup, and versioning
  Production builds must include centralized crash reporting, a semantic
  versionName plus incrementing Android versionCode, and a maintenance-state UI.
  Backend-originated history and balance data must be cached locally for
  read-only display during temporary backend outages while the platform restore
  process runs. The backend may enforce a minimum supported app version for
  forced upgrades when API compatibility changes.
  Historical minted audits are not silently recomputed when the biomass model
  changes.
  Backend-controlled satellite cutover also uses
  NISAR_PRODUCTION_READY=false by default; the mobile app has no local switch.

12. GEE quota behavior
  When backend satellite jobs are queued or quota-limited, the app continues
  polling the same audit_id and shows a delayed-processing message rather than
  retrying zone generation or sample submission aggressively. The client must
  keep the existing audit session alive and avoid starting a second
  /audit/zones or /audit/submit-samples request for the same parcel while the
  original audit_id is still active.

13. iOS future-scope blockers
   iOS requires an ARKit bridge, a BoundaryConfirmScreen base layer that does
   not depend on Android-only mapType="none", Apple-specific phone-auth setup,
   and separate Keychain validation before it can move into scope.

14. Audit due definition
  "Audit due" means once per calendar audit_year.
  If the user attempts to start another audit for a parcel that already has a
  current-year audit, the backend returns HTTP 409 with existing_audit_id and
  the app routes to status/resume instead of showing a generic error.

15. Android release minification
  Release APK/AAB builds require explicit ProGuard/R8 keep rules for ARCore,
  TensorFlow Lite, react-native-keychain, reanimated, and other native/reflective
  libraries used by the app. Validate at least one minified release build on a
  physical device before production rollout.

16. Audit certificate token mapping
  The backend derives the ERC-1155 certificate token ID using
  uuid.UUID(audit_id).int.
  The mobile app never computes this identifier locally.
